# Cookbook v1.2 — Design Spec (cart toggle highlight, rating on list cards)

**Status:** Ready to implement
**Date:** 2026-07-10
**Audience:** the implementing agent. This spec assumes the v1.1 codebase
(see [2026-07-10-v1.1-design.md](2026-07-10-v1.1-design.md)); file paths and
line anchors below are real and current as of the tip of `feature/v1.1`.

## 0. Summary of the two changes

1. **Cart button becomes a per-view toggle with a highlighted state.** On the
   recipe detail screen, each ingredient's "add to shopping list" button no
   longer shows a snackbar. Tapping it adds the currently-scaled line and
   **highlights that button**; tapping the highlighted button again **removes
   the exact item it added** and un-highlights. The highlight is ephemeral
   in-memory state that resets every time the screen is opened.
2. **Show the star rating on recipe list cards**, positioned **before** the
   tags. Unrated recipes show nothing.

### Resolved design decisions (do not re-litigate)
- **Feedback on add:** highlighted icon only — **remove the snackbar entirely**
  (and its Undo action).
- **Re-tapping a highlighted button:** **toggle off** — remove the item that
  button added and return the icon to its default state.
- **State source:** the highlight is **local `State` (per screen open), not a
  lookup against the shopping list.** Opening a recipe starts every button
  un-highlighted; there is **no scan** of the shopping list on open.
- **Serving scaler interaction:** changing servings **does not** rewrite or
  update anything already in the shopping list, and **does not** change any
  button's highlight. The added line keeps the amount it had when added.
- **List-card rating:** **unrated recipes show no stars** (not five empty
  outlines); rated recipes show filled/outline stars for their value, before
  the tags.

### Accepted consequence (by design, not a bug)
Because the highlight state resets when the detail screen is closed, the
toggle-off can only remove items added **during the current viewing**. Add an
item, leave, return, and tap again → a **second copy** is added; the earlier
one remains in the list untracked. Cleaning up duplicates is the user's job
(consistent with v1.1 §5: no de-duplication).

### Out of scope for v1.2 (note, don't build)
- Persisting the highlight across screen opens or deriving it from the actual
  shopping-list contents.
- Re-scaling / updating already-added shopping items when servings change.
- De-duplicating the shopping list.
- Rating shown anywhere other than the detail screen (already there) and the
  list card (this change).

## 2. Feature: cart button toggle + highlight (detail screen)

All changes are in `lib/src/ui/recipe_detail_screen.dart`, class
`_RecipeDetailScreenState`.

### 2.1 Add ephemeral state
Immediately after the existing `int? _targetServings;` field (currently
line 20), add:

```dart
  // Ingredient index → the shopping-list item id it added, for items added
  // during THIS viewing only. Ephemeral: this State (and the map) is recreated
  // empty every time the screen opens, so there is no lookup against the
  // shopping list on open. It is independent of the serving scaler, so
  // changing servings never rewrites an already-added line or a highlight.
  final Map<int, int> _addedItemIds = {};
```

### 2.2 Replace the add method with a toggle
Replace the whole `_addToShoppingList` method (currently lines 29–42,
snackbar-and-Undo version) with:

```dart
  Future<void> _toggleShoppingItem(
    int index,
    Ingredient ingredient,
    double factor,
  ) async {
    final repo = ref.read(shoppingRepositoryProvider);
    final existingId = _addedItemIds[index];
    if (existingId != null) {
      // Highlighted → remove the exact item this button added, un-highlight.
      await repo.removeItem(existingId);
      if (!mounted) return;
      setState(() => _addedItemIds.remove(index));
    } else {
      // Not added → add the currently-scaled line, highlight this button.
      final id = await repo.addItem(ServingScaler.scaledLine(ingredient, factor));
      if (!mounted || id == -1) return; // addItem returns -1 for empty text
      setState(() => _addedItemIds[index] = id);
    }
  }
```

There is **no snackbar** anywhere in this flow anymore.

### 2.3 Wire the index and the highlighted icon
Replace the ingredient loop (currently lines 216–239) with the indexed version
below. The change vs. current code: `.indexed` to get `index`, and the
`IconButton` icon/onPressed now reflect the toggle state.

```dart
                for (final (index, ingredient) in recipe.ingredients.indexed)
                  Row(
                    crossAxisAlignment: CrossAxisAlignment.center,
                    children: [
                      const Padding(
                        padding: EdgeInsets.only(right: 8),
                        child: Icon(Icons.circle, size: 6),
                      ),
                      Expanded(
                        child: Text(
                          ServingScaler.scaledLine(ingredient, factor),
                        ),
                      ),
                      IconButton(
                        visualDensity: VisualDensity.compact,
                        tooltip: _addedItemIds.containsKey(index)
                            ? 'Remove from shopping list'
                            : 'Add to shopping list',
                        icon: Icon(
                          _addedItemIds.containsKey(index)
                              ? Icons.shopping_cart // filled = added
                              : Icons.add_shopping_cart_outlined,
                          size: 20,
                          color: _addedItemIds.containsKey(index)
                              ? Theme.of(context).colorScheme.primary
                              : null,
                        ),
                        onPressed: () =>
                            _toggleShoppingItem(index, ingredient, factor),
                      ),
                    ],
                  ),
```

Icon contract (used by the tests): **not added** = `Icons.add_shopping_cart_outlined`;
**added/highlighted** = `Icons.shopping_cart` (filled) tinted with the theme
primary colour. Keep these two exact icons so `find.byIcon` in the tests can
distinguish states.

## 3. Feature: rating on recipe list cards

Change is in `lib/src/ui/recipe_list_screen.dart`, the `_RecipeCard` widget.

Replace the `ListTile`'s `subtitle:` (currently line 235):

```dart
        subtitle: recipe.tags.isEmpty ? null : Text(recipe.tags.join(' · ')),
```

with a stacked subtitle that puts the stars **before** the tags and renders
nothing when a recipe has neither:

```dart
        subtitle: (recipe.rating == null && recipe.tags.isEmpty)
            ? null
            : Column(
                crossAxisAlignment: CrossAxisAlignment.start,
                mainAxisSize: MainAxisSize.min,
                children: [
                  if (recipe.rating != null)
                    Row(
                      mainAxisSize: MainAxisSize.min,
                      children: [
                        for (var star = 1; star <= 5; star++)
                          Icon(
                            star <= recipe.rating!
                                ? Icons.star
                                : Icons.star_border,
                            size: 14,
                            color: Colors.amber,
                          ),
                      ],
                    ),
                  if (recipe.tags.isNotEmpty) Text(recipe.tags.join(' · ')),
                ],
              ),
```

Notes:
- Unrated (`recipe.rating == null`) → the star `Row` is omitted entirely (no
  empty outlines), matching the resolved decision.
- `star <= recipe.rating!` compares `int <= double`, which is valid in Dart and
  matches the detail screen's existing star logic.

### 4.1 `test/ui/recipe_detail_screen_test.dart`

  Keep the existing item-content assertion (`items!.single.text == '300 g Mehl'`).

Add a new test for **toggle-off** (single-ingredient recipe so `find.byIcon`
matches exactly one button):
```dart
  testWidgets('tapping the highlighted cart button removes the item it added', (
    tester,
  ) async {
    // (set up the same single-ingredient recipe + pump the detail screen)
    await tester.tap(find.byIcon(Icons.add_shopping_cart_outlined));
    await settle(tester);
    var items = await tester.runAsync(
        () => container.read(shoppingRepositoryProvider).watchItems().first);
    expect(items!, hasLength(1));

    await tester.tap(find.byIcon(Icons.shopping_cart)); // toggle off
    await settle(tester);
    items = await tester.runAsync(
        () => container.read(shoppingRepositoryProvider).watchItems().first);
    expect(items!, isEmpty);
    expect(find.byIcon(Icons.add_shopping_cart_outlined), findsOneWidget);
    expect(find.byIcon(Icons.shopping_cart), findsNothing);
    await flushTeardown(tester);
  });
```

Add a test that the **scaler does not disturb an added item or its highlight**:
add at 2 servings (`'200 g Mehl'`), then bump to 3 servings, and assert the list
still contains `'200 g Mehl'` (not `'300 g Mehl'`) and the icon is still
`Icons.shopping_cart`.

**Test-zone reminder:** awaiting a Drift **stream** (`watchItems().first`)
inside `testWidgets` hangs unless wrapped in `tester.runAsync(() => ...)` — the
existing cart test already does this; follow the same pattern.
