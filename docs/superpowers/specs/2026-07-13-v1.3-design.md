**Date:** 2026-07-13
**Audience:** anyone reviewing, extending, or porting this feature. File

## 0. Summary

Manually created and edited recipes can now group their ingredient list into
named sections (e.g. "Klopse" / "Sauce" for Königsberger Klopse) by starting a
line in the ingredients box with `#`. A `#`-line becomes a **section header**:
a bold subheading on the detail screen, excluded from serving scaling, the
shopping-cart button, and full-text search. Every other line is parsed as a
normal ingredient exactly as before.

This is a **manual-recipe-only** feature. Chefkoch imports (`RecipeParser`)
are untouched — imported recipes never contain section headers unless a user
edits them afterward and adds `#` lines by hand.

### Resolved design decisions
- **Marker:** a leading `#` on an otherwise-normal ingredient line, Markdown-heading
  style. One or more `#`s and any following whitespace are stripped; the rest
  of the line (trimmed) becomes the section name. `## Sauce`, `#Sauce`, and
  `#  Sauce ` all normalize to a header named `Sauce`.
- **Empty header:** a bare `#` with no heading text is **dropped entirely** —
  it becomes neither a section nor an ingredient.
- **Ingredients before the first header:** allowed and kept as an unlabeled
  leading group; there's no requirement that a recipe start with a header.
- **Validation:** a recipe needs **at least one real ingredient**; a form
  submission containing only `#` header lines (or none at all) fails
  validation with the existing "Add at least one ingredient" message.
- **Scaling:** section headers are **never** passed through
  `ServingScaler.scaledLine` — they carry no `amount`, so scaling doesn't
  apply to them and they aren't rendered through the scaled-line codepath at all.
- **Shopping list:** section headers get **no cart button** — they are not
  purchasable items.
- **Search (FTS):** section headers are **excluded** from the ingredients FTS
  column; only their name would ever appear in the query string.
- **Backup:** no `backup_service.dart` change was needed — it already
  round-trips full `Ingredient.toJson()`/`fromJson()` payloads, so
  `is_section` rides along automatically.
- **Backward compatibility:** `Ingredient.isSection` defaults to `false` and
  is **omitted from `toJson()` when false**, so every payload written before
  this feature (recipes, JSON-LD-imported ingredients, existing backups)
  decodes unchanged.

### Out of scope (not built)
- Chefkoch import never produces section headers (JSON-LD `recipeIngredient`
  is a flat string list; Chefkoch's own section structure, if any, is not
  parsed).
- No UI for inserting a `#` header other than typing it in the multi-line
  ingredients text box (no button, no per-line menu).
- No collapse/expand of sections on the detail screen.
- No section-aware shopping list (adding "all of a section" at once).
- No migration of existing recipes — nothing to migrate, since sections are
  purely additive to the ingredient JSON shape.

## 1. Data model

`lib/src/models/recipe.dart` — `Ingredient` gained one field:

```dart
final bool isSection; // default false
```

- `Ingredient.section(String name)` factory: builds a header with
  `name: name`, `raw: '# $name'`, `isSection: true`. The `raw` form is what
  the manual-entry form round-trips back into the ingredients text box in
  edit mode (`widget.existing?.ingredients.map((i) => i.raw).join('\n')`),
  so re-editing a recipe with sections shows the original `#` lines again.
- `toJson()` only writes `'is_section': true` when `isSection` is true —
  omitted (not `false`) otherwise, keeping pre-v1.3 payloads byte-identical.
- `fromJson()` reads `json['is_section'] as bool? ?? false` — absent key
  decodes as `false`, so old data (and old backups) load fine.
- `==`/`hashCode`/`toString()` all include `isSection`.

No schema version bump was needed anywhere: `ingredients_json` is an opaque
TEXT blob in Drift (`Recipes.ingredientsJson`), and the recipe payload
`kCurrentSchemaVersion` (currently 1) governs blob *shape* compatibility, not
individual field presence — an additive, defaulted JSON key doesn't break it.

## 2. Parsing `#` lines in the manual form

`lib/src/manual/manual_recipe.dart` adds a private helper that runs instead of
the old flat `parseIngredient` map:

```dart
final RegExp _sectionMarker = RegExp(r'^#+\s*');

List<Ingredient> _parseIngredientLines(String text) {
  final result = <Ingredient>[];
  for (final line in _nonEmptyLines(text)) {
    if (line.startsWith('#')) {
      final name = line.replaceFirst(_sectionMarker, '').trim();
      if (name.isNotEmpty) result.add(Ingredient.section(name));
    } else {
      result.add(parseIngredient(line));
    }
  }
  return result;
}
```

`buildManualRecipe` calls `_parseIngredientLines(ingredientsText)` in place of
the old `for (line in _nonEmptyLines(...)) parseIngredient(line)` list
comprehension. Behavior:
- `_nonEmptyLines` already trims and drops blank lines before this runs.
- Any line starting with `#` is a header candidate; `_sectionMarker` strips
  all leading `#`s and whitespace in one match (`^#+\s*`), so `##  Sauce ` →
  strip `"##  "` → `"Sauce "` → trim → `"Sauce"`.
- A line that's `#` alone (or `#` + only whitespace) strips to an empty
  string and is dropped — not added as a header, not added as an ingredient.
- Everything else goes through the existing German-aware `parseIngredient`
  unchanged.

## 3. Detail screen rendering

`lib/src/ui/recipe_detail_screen.dart`, in the ingredients loop (was a single
`Row` per ingredient; now branches on `ingredient.isSection`):

```dart
for (final (index, ingredient) in recipe.ingredients.indexed)
  if (ingredient.isSection)
    Padding(
      padding: const EdgeInsets.only(top: 14, bottom: 4),
      child: Text(
        ingredient.name,
        style: Theme.of(context).textTheme.titleMedium
            ?.copyWith(fontWeight: FontWeight.bold),
      ),
    )
  else
    Row(/* unchanged: bullet, ServingScaler.scaledLine(ingredient, factor), cart IconButton */),
```

- A section header renders as a bold `titleMedium` line with extra top
  padding (visual separation from the previous group) — no bullet, no
  scaled-amount text, no cart button.
- Every non-section row is exactly the v1.2 cart-toggle row (see the v1.2
  spec) — `index` here is the recipe-wide ingredient index, so the
  `_addedItemIds` per-view highlight map (keyed by that same index) still
  addresses the right row even with headers interleaved.

## 4. Search: sections excluded from the FTS index

`lib/src/repository/local_recipe_repository.dart`, `_writeFtsRow`:

```dart
[
  id,
  title,
  tags.join(' '),
  // Section headers (v1.3) are display-only labels, not searchable
  // ingredients — keep them out of the FTS index.
  ingredients.where((i) => !i.isSection).map((i) => i.name).join(' '),
],
```

The only change is the `.where((i) => !i.isSection)` filter before joining
ingredient names into the FTS `ingredients` column. Titles and tags are
unaffected. A recipe search for a section's own name (e.g. "Klopse" as a
header, distinct from any ingredient literally named Klopse) will not surface
that recipe via the header text — only via title, tags, or a real ingredient.

## 5. Form validation

`lib/src/ui/recipe_form_screen.dart` replaces the ingredients field's
validator. Previously any non-empty line satisfied
`_requiredLines(v, 'Add at least one ingredient')`; now:

```dart
String? _requireIngredient(String? value) =>
    (value ?? '').split('\n').any((l) {
      final line = l.trim();
      return line.isNotEmpty && !line.startsWith('#');
    })
    ? null
    : 'Add at least one ingredient';
```

A submission with only `#` lines (or blank lines) fails with the same
"Add at least one ingredient" message as an empty box — sections don't count
as content on their own.

The field's hint/help text was also updated to show the new syntax:

```dart
hintText: '# Klopse\n500 g Hackfleisch\n2 Eier\n# Sauce\n40 g Butter',
helperText: 'Start a line with # to add a section header (e.g. # Sauce).',
```

## 6. Testing

All added tests follow the existing per-layer TDD pattern; look here for the
exact assertions to preserve if this feature is touched again.

- **`test/models/recipe_test.dart`** — `Ingredient.section` round-trips
  through `toJson`/`fromJson`; `toJson()` omits `is_section` when false; a
  legacy payload with no `is_section` key decodes as `isSection: false`.
- **`test/manual/manual_recipe_test.dart`** (group `buildManualRecipe (#
  sections)`) — `#` lines interleaved with ingredients split correctly and
  preserve order; `##  Sauce ` normalizes to name `Sauce` / `raw: '# Sauce'`;
  a bare `#` is dropped (`ingredients` length unaffected); ingredients before
  the first `#` stay as an unlabeled leading group.
- **`test/repository/local_recipe_repository_test.dart`** — a real
  ingredient in a sectioned recipe is findable by FTS search; the section
  header's own text is not.
- **`test/ui/recipe_detail_screen_test.dart`** — a two-section recipe
  (`# Klopse` / real ingredient / `# Sauce` / real ingredient) shows both
  header texts and exactly two cart buttons (not four).
- **`test/ui/recipe_form_screen_test.dart`** — a recipe of only `#` lines
  fails validation and saves nothing; a `#` header plus one real ingredient
  saves successfully with `ingredients.first.isSection == true`.
