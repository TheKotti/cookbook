# Cookbook v1.1 — Design Spec (ratings, no back-link, local pictures, shopping list)

**Status:** Ready to implement
**Date:** 2026-07-10
**Audience:** the implementing agent. This spec assumes the v1 codebase described in
[DESIGN.md](../../../DESIGN.md); file paths below are real and current.

## 0. Summary of the four changes

1. **Editable ratings, no imported ratings.** Stop importing Chefkoch's
   `aggregateRating`. Let the user set a 1–5 whole-star rating per recipe, editable
   inline on the detail screen. Existing imported ratings are left in place (harmless)
   and are overwritten the moment the user taps a star.
2. **Remove the Chefkoch back-link** from the UI. The detail screen must never navigate
   back to chefkoch.de. `sourceUrl` stays in the model/DB (it's the dedup key and the
   `isManual` marker) but is never surfaced.
3. **Manually added pictures.** Pick from gallery or camera; copy the file into app
   storage; store a local path on the recipe. Local image takes precedence over any
   imported `imageUrl` when rendering.
4. **Shopping list (new view).** A cart button on each ingredient adds its *currently
   displayed (scaled)* line to a single global shopping list. A new Shopping List screen
   (reached via a cart icon + count badge in the recipe-list app bar) lets the user
   check items off (which removes them), edit them, and add items manually.

### Resolved design decisions (do not re-litigate)
- Rating scale: **1–5 whole stars**; tapping the current value clears it (null).
- Picture source: **gallery + camera** via `image_picker`, copied into app documents.
- Cart adds: **the scaled line as shown** (honors the serving scaler).
- Shopping entry point: **cart icon with an item-count badge** in the recipe-list app bar.

### Out of scope for v1.1 (note, don't build)
- Combining/merging duplicate shopping items or summing quantities.
- Embedding image bytes in the JSON backup (only the path is backed up — see §4.6).
- Per-recipe shopping lists, rating on list cards, and anything cloud (§12 of DESIGN.md).

## 1. Version & migration overview

There are **two independent version concepts** — don't conflate them:

- **Drift DB `schemaVersion`** (`lib/src/db/database.dart`, currently `1`): governs SQLite
  migrations. **Bump to `2`** for the new column + new table.
- **Recipe payload `kCurrentSchemaVersion`** (`lib/src/models/recipe.dart`, currently `1`):
  governs the shape of the ingredients/steps JSON blobs. **Leave at `1`** — the blob
  shapes don't change.

Drift migration (`MigrationStrategy`):
- `onCreate`: unchanged plus the new table (or just use `m.createAll()` which already
  creates all declared tables) — keep the existing FTS `customStatement`.
- Add `onUpgrade`:
  ```dart
  onUpgrade: (m, from, to) async {
    if (from < 2) {
      await m.addColumn(recipes, recipes.localImagePath);
      await m.createTable(shoppingItems);
    }
  }
  ```

After any change to `database.dart`, **regenerate** `database.g.dart`:
`dart run build_runner build --delete-conflicting-outputs` (see README).

## 2. Feature: editable ratings, no import

### 2.1 Stop importing ratings
`lib/src/parser/recipe_parser.dart`
- Remove the `rating:` argument's use of `_rating(...)` in `parse(...)` — set
  `rating: null` (or drop the named arg so it defaults to null).
- Delete the now-unused `_rating` helper.
- Update `test/parser/recipe_parser_test.dart`: any assertion on a parsed `rating` now
  expects `null`.

Existing DB rows keep their old rating values; that's acceptable per requirements. Do
**not** write a migration to null them out.

### 2.2 Persist an edited rating
`lib/src/repository/recipe_repository.dart` — add to the interface:
```dart
Future<void> setRating(int recipeId, double? rating);
```
`lib/src/repository/local_recipe_repository.dart` — implement it in the same style as
`setTags`: update the row's `rating`, and **touch the row** so table watchers re-emit.
Rating is **not** in the FTS index, so do **not** rewrite the FTS row here.
```dart
Future<void> setRating(int recipeId, double? rating) {
  return (db.update(db.recipes)..where((r) => r.id.equals(recipeId)))
      .write(RecipesCompanion(rating: Value(rating)));
}
```
(A single `write` already causes watchers to re-emit; no separate touch needed.)

### 2.3 UI — interactive stars on the detail screen
`lib/src/ui/recipe_detail_screen.dart`
- Remove the read-only rating `_InfoItem` (the `if (recipe.rating != null)` star item in
  the `Wrap` at ~L154–160).
- Add an always-present 1–5 star row (a small private widget) near the title/info area:
  - Render 5 tappable stars; filled (`Icons.star`) for `i <= rating`, outline
    (`Icons.star_border`) otherwise. Treat `rating == null` as 0 filled.
  - Tap star `i` → if `recipe.rating == i` call `setRating(id, null)` (clear); else
    `setRating(id, i.toDouble())`.
  - It reads from the watched `recipe` and writes via
    `ref.read(recipeRepositoryProvider).setRating(...)`; the stream refresh updates it.
- No change to the edit form for ratings; `buildManualRecipe` already preserves
  `existing.rating`, and new manual recipes start unrated (null), which the star row
  handles.

## 4. Feature: manually added pictures (local files)

### 4.1 Dependency
`pubspec.yaml`: add `image_picker` (latest stable). `path_provider` and `path` are
already present.

### 4.2 Model
`lib/src/models/recipe.dart`
- Add field `final String? localImagePath;` (a **relative** filename, resolved against
  the app images dir at render time — see §4.4). Add to the constructor, `toJson`
  (`'local_image_path'`), `fromJson`, and extend `copyWith` to carry it (currently
  `copyWith` only handles `id` and `tags`; add an optional `localImagePath`).

### 4.3 DB
`lib/src/db/database.dart`
- `Recipes` table: add `TextColumn get localImagePath => text().nullable()();`
- Migration + regenerate as in §1.
`lib/src/repository/local_recipe_repository.dart`
- Include `localImagePath: Value(recipe.localImagePath)` in the `RecipesCompanion` in
  `saveRecipe`, and map `row.localImagePath` in `_toModel`.

### 4.4 Image storage service (new)
`lib/src/images/image_store.dart` — small, testable, filesystem-only:
- `Future<String> save(String sourceFilePath)`: copy the picked file into
  `<appDocumentsDir>/images/`, filename `${DateTime.now().microsecondsSinceEpoch}<ext>`,
  return the **relative** path (e.g. `images/168...jpg`). No new uuid dependency needed.
- `File resolve(String relativePath)`: join against `getApplicationDocumentsDirectory()`.
- `Future<void> delete(String relativePath)`: best-effort remove; ignore if missing.
- Expose via a Riverpod provider in `providers.dart` (`imageStoreProvider`).

Store relative paths (not absolute) so they survive app-data-dir changes across updates.

### 4.5 UI
`lib/src/ui/recipe_form_screen.dart` (used for both create and edit)
- Add an image section at the top: a preview (local file if set, else the imported
  network image, else a placeholder) plus buttons **Add photo / Change / Remove**.
- **Add/Change** opens a small chooser (bottom sheet): *Take photo* → `ImagePicker`
  with `ImageSource.camera`; *Choose from gallery* → `ImageSource.gallery`. On pick,
  call `imageStore.save(picked.path)`, hold the returned relative path in form state.
- **Remove** clears the local path in form state (optionally delete the old file on save).
- Thread the value through `buildManualRecipe` (`lib/src/manual/manual_recipe.dart`): add
  a `String? localImagePath` param; in edit mode default to `existing?.localImagePath`
  when the form didn't change it, and set the new value when it did. Preserve/override
  semantics mirror how `imageUrl`/`rating` are currently preserved.

Rendering helper (shared): add a single widget/function that, given a recipe, returns the
right image: **local file first** (`Image.file(imageStore.resolve(path))`), then
`imageUrl` (`CachedNetworkImage`), then the `Icons.restaurant` placeholder. Use it in:
- `recipe_detail_screen.dart` header image (~L95–108), and
- `recipe_list_screen.dart` `_RecipeCard` leading thumbnail (~L213–226).

### 4.6 Deletion cleanup & backup
- When a recipe is deleted, delete its local image file too. `deleteRecipe` in the
  repository is storage-only; do the file cleanup in the caller
  (`_confirmDelete` in the detail screen): read `recipe.localImagePath` before deleting,
  then `imageStore.delete(...)` after.
- `lib/src/backup/backup_service.dart`: add `'local_image_path'` to `_toJson` and read it
  in `_fromJson` (tolerate missing → null). **Keep `format_version: 1`** — adding an
  optional field is backward-compatible, and the reader already tolerates absent fields.
  Document the known limitation: the **image file is not embedded**, only its path, so a
  backup restored on a different device shows a placeholder for local images. (Same
  spirit as imported `image_url` today.)

### 4.7 Platform permissions
- Android: `image_picker` generally needs no extra manifest permission for gallery/camera
  via intents; verify a build works. If a device requires it, add `CAMERA` to
  `android/app/src/main/AndroidManifest.xml`.
- iOS (for the future iOS target): add `NSCameraUsageDescription` and
  `NSPhotoLibraryUsageDescription` to `ios/Runner/Info.plist`.

## 5. Feature: shopping list

One **global** shopping list. Items are free-text strings. Checking an item **removes**
it (no persisted "done" state). Items can be edited and added manually.

### 5.1 Model (new)
`lib/src/models/shopping_item.dart`
```dart
class ShoppingItem {
  final int id;
  final String text;
  final DateTime createdAt;
  const ShoppingItem({required this.id, required this.text, required this.createdAt});
}
```

### 5.2 DB (new table)
`lib/src/db/database.dart`
```dart
class ShoppingItems extends Table {
  IntColumn get id => integer().autoIncrement()();
  TextColumn get text => text()();
  TextColumn get createdAt => text()(); // ISO-8601
}
```
Add to `@DriftDatabase(tables: [...])`. Created in `onCreate` via `m.createAll()` and in
`onUpgrade` via `m.createTable(shoppingItems)` (§1). Order lists by `id ASC` (insertion
order).

### 5.3 Repository (new, separate seam)
`lib/src/repository/shopping_repository.dart` (interface) and
`lib/src/repository/local_shopping_repository.dart` (Drift impl):
```dart
abstract class ShoppingRepository {
  Stream<List<ShoppingItem>> watchItems();
  Future<int> addItem(String text);      // trims; ignores empty
  Future<void> updateItem(int id, String text);
  Future<void> removeItem(int id);
  Future<void> clearAll();
}
```
`providers.dart`: add `shoppingRepositoryProvider` and
`shoppingItemsProvider = StreamProvider<List<ShoppingItem>>`.

### 5.4 Cart button on ingredients
`lib/src/ui/recipe_detail_screen.dart`
- In each ingredient row (the `for (final ingredient in recipe.ingredients)` loop,
  ~L207–224), add a trailing `IconButton(icon: Icon(Icons.add_shopping_cart_outlined))`.
- onPressed: `addItem(ServingScaler.scaledLine(ingredient, factor))` — the **same scaled
  string** already rendered in the row, so the shopping amount matches the current
  serving selection. Show a `SnackBar('Added to shopping list')` (optionally with an
  Undo action calling `removeItem`).

### 5.5 Shopping list screen (new view)
`lib/src/ui/shopping_list_screen.dart`
- AppBar `Shopping list`; overflow menu with **Clear all** (confirm dialog → `clearAll`).
- Body: watch `shoppingItemsProvider`.
  - Each row: a leading `Checkbox` (always unchecked) whose `onChanged` → `removeItem(id)`
    ("check off = remove"). Tapping the row text opens an **edit dialog** (TextField
    prefilled) → `updateItem(id, newText)`. (Swipe-to-dismiss may also call `removeItem`
    as a secondary gesture — optional.)
  - Empty state: "Your shopping list is empty. Add items from a recipe, or type one below."
- A persistent bottom input row: `TextField` + Add button → `addItem(text)`, clears the
  field, keeps focus for rapid entry.

### 5.6 Entry point — app-bar cart icon with badge
`lib/src/ui/recipe_list_screen.dart`
- Add an `IconButton` (`Icons.shopping_cart_outlined`) to the app-bar `actions` (before
  the existing backup `PopupMenuButton`). Wrap it in a Material 3 `Badge` showing the
  item count from `shoppingItemsProvider` (hide the badge when count is 0).
- onPressed: push `MaterialPageRoute(builder: (_) => const ShoppingListScreen())`.

Duplicates: adding the same ingredient twice appends two rows (no merge) — acceptable for
v1.1.

## 7. Testing

Follow the existing TDD style (see `test/`). At minimum:
- **Parser:** `recipe_parser_test.dart` — parsed `rating` is now `null`.
- **Recipe repo:** `setRating` sets and clears; `localImagePath` round-trips through
  `saveRecipe`/`getRecipe`.
- **Shopping repo:** add/update/remove/clear + `watchItems` ordering (insertion order).
- **Shopping screen:** checkbox removes; edit dialog updates; bottom input adds; empty
  state renders.
- **Detail screen:** no `chefkoch.de` link; tapping a star persists via `setRating`;
  tapping the current rating clears it; cart button adds the scaled line.
- **Form screen:** picking an image stores a local path on save; Remove clears it.
  (Fake/inject the `ImagePicker` + `ImageStore` so tests don't touch a real camera/FS.)
- **Backup:** `local_image_path` round-trips; a v1 backup lacking the field imports fine.
- **Image store:** `save` copies + returns a relative path; `resolve`/`delete` behave.

Keep `RecipeParser`, `ImageStore`, and the repositories pure/injectable so tests avoid
platform channels. Run `flutter test` and `flutter analyze` green before finishing.

## 8. Suggested build order

1. **DB & model plumbing:** add `localImagePath` column, `ShoppingItems` table, bump
   `schemaVersion` to 2 + `onUpgrade`, regenerate `database.g.dart`. Model + repo fields.
2. **Ratings:** stop importing (parser), add `setRating`, star widget on detail. Tests.
3. **Remove back-link:** detail screen + drop `url_launcher`. Tests.
4. **Image store + pictures:** `image_store.dart`, form UI, shared render helper, delete
   cleanup, backup field, permissions. Tests.
5. **Shopping list:** model, repo, providers, cart buttons, screen, app-bar badge. Tests.
6. **Verify:** `flutter analyze` + `flutter test`; build/run on Android and exercise each
   flow (rate a recipe, add a photo, add ingredients to the list, check them off).
